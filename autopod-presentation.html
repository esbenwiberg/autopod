🔄 Validation Feedback Loop

When tests fail, Autopod doesn't give up. It feeds the exact failure context back to the agent — which tests failed, what was expected, what happened — and lets it retry in the same session. Up to 3 correction attempts before escalating to a human.

🛠 Auto Fix Sessions on CI Failure

Even after merge, if CI breaks, Autopod spawns a new fix session automatically. It reads the CI logs, understands the failure, fixes the code, and re-opens a PR. No human needs to triage the failure. Up to 3 fix attempts, configurable.

💬 Human Rejection → Fresh Run

When a human rejects work (not just validation), the attempt counter resets completely. The agent gets fresh context, the human's feedback, and starts again from that state. Not a new session — a smarter retry.

Quality Gates

Four-phase validation before any merge

Not just "does it compile." Every PR runs a full verification pipeline, inside the same container the agent worked in.

Phase 01
Build
Runs your profile's build command. Captures stdout, stderr, exit code. No assumptions.
Phase 02
Health Check
Polls your health check URL until 200 OK or timeout. Configurable per profile.
Phase 03
Playwright Smoke Tests
Auto-generated browser tests run against your app. Screenshots captured and embedded in PRs.
Phase 04
AI Task Review
The diff + acceptance criteria go to an AI reviewer. Pass/fail with specific notes fed back to agent.

↩ Failures at any phase produce structured feedback, automatically sent back to the agent for correction. Findings are fingerprinted — duplicates across retries don't cause extra loops.

Human in the Loop

Agents that know when to ask

Autopod agents aren't fire-and-forget. When they hit an ambiguous requirement, they block and wait. When they need credentials, they request them — and the daemon injects them directly into the container without exposing the token. After a configurable threshold, the session auto-pauses for human review.

● ask_human — blocks session until human responds (up to 1hr timeout)
● ask_ai — consults a second AI model, rate-limited to 5 per session
● report_blocker — auto-pauses after N blockers, configurable threshold
● request_credential — credential injected by daemon, agent never sees the token
● validate_in_browser — Playwright smoke test triggered by the agent itself
escalation-mcp / tools / ask_human.ts 
// Injected MCP server inside agent container
// Agent calls this when it needs human input

const ask_human = tool({
  description: 'Block until human responds',
  inputSchema: {
    question: string,
    context?: string,
    timeout_ms?: number   // default: 3_600_000
  },
  async handler({ question, context, timeout_ms }) {
    // Push escalation to daemon event bus
    const pending = await bridge
      .createEscalation({
        type: 'ask_human',
        question, context
      });

    // Block until human responds or times out
    return await pending.waitForResolution(
      timeout_ms ?? 3_600_000
    );
  }
});

🛂 Policy Gate — Three Layers

Every action is checked against the profile's ActionPolicy before execution. Three independent checks must all pass:

Group enabled? — action group must be in the profile's allowlist
Approval gate — high-risk actions require explicit human approval before execution
Resource restriction — per-action allowlist of specific repos, orgs, or endpoints
Per-action override — disable or reconfigure any built-in action per profile

🧹 3-Stage Sanitization Pipeline

Every response from a backend passes through three sequential filters before the agent sees a single byte:

① Prompt Injection Quarantine
Score-based detection. Above threshold → blocked entirely before any processing.
② PII Stripping
Emails, API keys, AWS/Azure secrets, IP addresses stripped via regex patterns.
③ Field Whitelist
Only explicitly configured fields returned. Everything else redacted — including surprise fields from API updates.

⚙️ Fully Configurable Per Profile

ACP isn't a global toggle. Every profile configures its own action surface:

ActionPolicy 
{
  enabledGroups: [
    'github-issues',
    'ado-workitems'
  ],

    preset: 'strict'

Security Model

Defense in depth, not defense in hope

Credentials never leave the daemon. Agents can't reach the internet without permission. Every action is logged. Every output is sanitized.

🔒
Network Firewall — Three Tiers

Every container gets its own network stack. You choose the security posture per profile. The restricted mode uses dnsmasq with wildcard host matching — fine-grained enough for package registries and API endpoints, strict enough for sensitive workloads.

allow-all
unrestricted outbound — for trusted agents
deny-all
zero outbound — pure sandbox mode
restricted
allowlist of host:port pairs with wildcard support
🗝
AES-256 Credential Encryption

All stored credentials are AES-256 encrypted at rest. The key lives at ~/.autopod/secrets.key, separate from the database. Agents receive credentials via daemon injection — the plaintext token never appears in logs or in the container filesystem.

🧹
PII Sanitization Pipeline

Agent output passes through a pattern-detection engine before storage. Matches against known PII patterns and prompt injection attempts. Quarantines suspicious output with flagged sections before persisting to the database.

🔑
HMAC Session Tokens

Session tokens are HMAC-signed, short-lived, and validated on every request. In production, Azure Entra ID authentication is enforced end-to-end. Development mode stubs auth for local testing without config overhead.
