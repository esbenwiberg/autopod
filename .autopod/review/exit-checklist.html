ADRs introduced (1)

ADR-017 Screenshots on disk with retention Accepted

Decision

Move proof-of-work screenshots out of the SQLite blob column and onto disk at <dataDir>/screenshots/<podId>/<source>/<filename>.png. Daemon serves them via GET /pods/:podId/screenshots/.... Cutover migration drops the column with no backfill. Per-pod terminal-state retention sweep with default 30-day TTL.

Consequences

DB stops growing with screenshot data; backups become small again.
Reviewers see full resolution from desktop (lightbox); thumbnails fetch lazily.
ADO PRs get real attachment URLs instead of dead GitHub links.
Pre-cutover pods lose their UI screenshots permanently (drop-on-cutover).
Two new on-disk locations and one new HTTP route to operate, monitor, back up.
Mandatory pre-migration DB snapshot — reversibility cost.
Loopback-only screenshot endpoint — daemon trust boundary, not a public asset host.
30-day default retention; configurable via AUTOPOD_SCREENSHOT_RETENTION_DAYS.

Alternatives rejected

Keep base64 in DB, add lightbox. Solves the UX but not the growth — DB hits multi-GB territory in months.

Object storage (S3/Azure Blob). Adds external dependency for a single-user local tool. Not justified at current scale.

Source: docs/decisions/ADR-017-screenshots-on-disk-with-retention.md
