# Project Facts

Lightweight, queryable decision records for use by planning agents.

Facts capture durable preferences, conventions, and policies that recur across
features. They are lighter than ADRs — no status lifecycle, no ceremony — but
similarly queryable. Where an ADR records a hard-to-reverse architectural
decision, a fact records a soft, recurring preference that would otherwise
prompt the same question again on the next feature.

## When to write a fact

Write a fact when a planning interview surfaces an answer that:
- Would apply to the next feature too (not feature-specific)
- Is not surprising or hard-to-reverse enough to warrant an ADR
- Would save a planning agent from asking the same question again

Examples of facts:
- "New API endpoints live under `/api/v2`, not `/api/`"
- "Feature flags are preferred over version branches for gradual rollout"
- "All new daemon packages must be added to the pnpm workspace"
- "Migration files use a numeric prefix; never reuse a prefix"

Do NOT write a fact for:
- A decision specific to one feature → put it in that spec's `design.md`
- A hard-to-reverse or surprising architectural call → write an ADR instead
- Something obvious from the codebase structure → the agent will find it

## Format

One file per fact: `fact-NNN-<short-slug>.md`, numbered globally and monotonically.

```markdown
---
topics: [api, deployment]
---

# Title — a plain-English statement of the fact

One to three sentences: the rule itself, then the rationale in one sentence.
No headers, no bullet lists. Dense is good.
```

Front matter fields:
- `topics` — one or more coarse categories from the taxonomy below. Required.

Body:
- H1 title: the fact as a short declarative statement
- Body: rule + rationale. No subheadings. Keep it to 3 sentences max.
  The planning agent reads this inline — density matters.

## Topics taxonomy

Keep topics coarse and stable. Pick from this list; add a new entry only when
nothing fits — don't fragment.

| Topic | Covers |
|-------|--------|
| `api` | HTTP routes, REST conventions, versioning, response shapes |
| `auth` | Authentication, authorization, tokens, identity providers |
| `data` | Database, migrations, schema conventions, storage |
| `deployment` | Docker, ACI, env vars, startup, health checks |
| `security` | Encryption, secrets, network isolation, audit |
| `testing` | Test patterns, mocking, test infra, coverage expectations |
| `tooling` | Build, lint, package manager, workspace conventions |
| `workflow` | Branching, PRs, commit conventions, review gates |

## Index

`index.md` is auto-generated by `scripts/generate-knowledge-index.sh`.
Run it after adding or changing any fact. Never edit `index.md` by hand.
`docs/decisions/index.md` is generated by the same script.

## Lifecycle

Facts are not append-only. If a fact becomes stale or is superseded by an ADR,
delete it or update it. The index regenerates cleanly from whatever files remain.
There is no "Superseded" status — stale facts are just removed.
