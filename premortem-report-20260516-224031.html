Premortem report

ACs to Durable Facts

Premise: it is 6 months from now, Autopod moved from acceptance criteria as temporary pod gates to executable facts as repo memory, and the effort failed.

6 failure agents Autopod specs Facts-first workflow Generated 2026-05-16 22:40
Most Likely Failure

Facts become a new brief section, not a merge-enforced repo artifact.

Autopod already has several AC surfaces. Adding vocabulary is easier than changing merge behavior.

Most Dangerous Failure

Low-quality facts create false confidence.

A bad test can preserve implementation details while missing the behavior the original AC cared about.

Hidden Assumption

Executable does not automatically mean durable truth.

A fact needs CI enforcement, behavior ownership, and intent-preserving review.

Revised Plan
  1. Define one canonical contract: ACs are pre-merge probes, test expectations are package-owned tests, durable facts are merge-enforced artifacts, and human-review-only is an explicit exception.
  2. Use Given/When/Then only for multi-step behavior, state machines, workflows, and ambiguous rules.
  3. Require every behavioral AC to map to a changed durable artifact or a human-review-only exception.
  4. Prefer co-located package tests over external probes for state machines, parsers, repositories, validation, profiles, and CLI behavior.
  5. Pilot on daemon behavior briefs before expanding to desktop, integrations, and ops workflows.
Single Most Important Revision

Make “durable fact” a merge rule, not a writing style.

The pod should list artifact paths or explicitly explain why a criterion remains human-review-only.

Pre-Launch Checklist

  • Update `/prep`, `/plan-feature`, CLI/issue watcher parsing, desktop labels, and validation docs with one vocabulary.
  • Add a final-summary or PR requirement mapping each behavioral AC to a durable artifact path or `human_review_only` reason.
  • Reject generic `run tests`, snapshots, fixture-only assertions, and private-structure tests as durable facts.
  • Measure 10 pods: brief size, prep time, strengthened tests, human-review-only exceptions, and weak-fact review rejections.
  • Start with daemon/package tests before expanding to desktop and external integrations.

Failure Modes

High likelihoodHigh severity

1. Renamed ACs Without a Merge Gate

“Facts-first” exists mostly as vocabulary. Briefs and PR bodies mention durable facts, but merges still proceed when mappings are aspirational.

Assumption: better structure changes merge behavior without an enforceable gate.

Warning signs: fact mappings in prose with no changed test or invariant; reviews accept verbal confirmation.

Medium likelihoodMedium severity

2. Ceremony Without Better Tests

Every AC blooms into Given/When/Then, durable mapping, exceptions, and validation notes. Briefs get heavier while tests stay flat.

Assumption: explicit prose naturally becomes durable tests.

Warning signs: brief size and prep time rise 30%+, but strengthened tests per pod do not.

High likelihoodHigh severity

3. Facts Live Too Far From Code

`api`, `web`, and `cmd` probes catch surface behavior but miss the lower-level invariants that keep the repo sane.

Assumption: external validation probes can stand in for behavior-owned tests.

Warning signs: new probes land without changes to co-located `*.test.ts` files.

Medium likelihoodHigh severity

4. Low-Quality Facts Create False Memory

Agents generate brittle snapshots, mock call-order assertions, and fixture checks that satisfy the shape of a fact without protecting intent.

Assumption: agents reliably translate behavior into meaningful tests without quality gates.

Warning signs: tests assert DOM structure, private helpers, or exact blobs and rarely fail under seeded regressions.

Medium likelihoodMedium severity

5. Boundaries Resist Simple Fact Conversion

Desktop UX, external integrations, auth, notifications, and ops workflows do not fit cleanly into stable repo-local checks.

Assumption: most meaningful behavior can be translated into repo-owned executable facts without losing context.

Warning signs: more than 25-30% of behavioral ACs become human-review-only or fake checks.

High likelihoodMedium severity

6. Split-Brained Schema and Tooling

Briefs, desktop forms, issue watcher parsing, CLI `--ac-from`, validation, and skills adopt different meanings for AC, scenario, fact, and test expectation.

Assumption: better terms converge the ecosystem without one canonical contract at every boundary.

Warning signs: the same requirement round-trips as AC, scenario, fact, and test expectation with semantic loss.
